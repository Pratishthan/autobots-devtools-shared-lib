02 Topology — one core, two consumers

The valuable part of your Chainlit apps is the Python agent logic underneath, which is framework-agnostic LangChain. Factor it into a shared package so nothing is rewritten and nothing is thrown away — existing Chainlit apps keep running, Atlas gets a dedicated service, both import the same core.

What ports from Chainlit: tool bodies, prompts, model config copy over directly. Your cl.Step shape (name · input · output · status) maps almost 1:1 onto the Atlas tool-step event below — it's the best template you already have for the contract. The cl.LangchainCallbackHandler pattern becomes the LangGraph-stream → SSE translator.

03 Authentication — GitHub OAuth

GitHub OAuth2 Authorization Code flow with the backend as a confidential client. Authentication alone isn't enough for an internal tool, so access is gated on org membership and GitHub teams map to Atlas roles. The GitHub token never reaches the browser — the backend issues its own session cookie.

UI starts login
GET /auth/login → 302 to GitHub authorize with scope=read:org user:email and a signed state.
GitHub redirects back
GET /auth/callback?code=&state= — backend verifies state, exchanges code for a token server-side.
Authorize, not just authenticate
Fetch /user + org membership; reject anyone outside the org. Resolve the user's team(s).
Issue session
Map GitHub team → Atlas role, mint an httpOnly · Secure · SameSite signed cookie. GitHub token stays in the secrets store, never in the browser.
Every call carries identity
/chat and /approvals require the cookie; the resolved actor (GitHub login) is stamped on the run — feeds the approval audit trail.

Signed cookie, httpOnlyBackend confidential client · token server-side only
Gate

↻

Chainlit already does this. Chainlit ships native GitHub OAuth via @cl.oauth_callback with OAUTH_GITHUB_CLIENT_ID / OAUTH_GITHUB_CLIENT_SECRET. Put the org-check + team→role mapping in atlas_core.auth and reuse one GitHub OAuth App (two callback URLs: Chainlit + Atlas) so both surfaces share a single identity policy and secret.

04 Request lifecycle

Client opens the stream
POST /chat (session cookie required) with {conversation_id, message} → server responds text/event-stream.
Agent reasons & emits tokens
LangGraph streams the assistant message as token events; UI appends them live.
Tool call starts
tool_step (phase:"start") opens a collapsible step in the thread.
Gated tool pauses for approval
For run_python / query_warehouse, the graph hits interrupt_before, checkpoints, and emits approval_request. The stream stays open.
Human decides
POST /approvals/{id} with approve / reject (and optional edited input). Graph resumes from the checkpoint.
Tool resolves
tool_step (phase:"end") with output + status; agent continues or finishes.
Turn ends
done with message_id + finish_reason; server closes the stream.

05 SSE event contract

Every server→client frame is one SSE data: line carrying a JSON object with a type discriminator. Client→server actions are plain JSON POSTs. This is the surface the UI and backend must agree on.

tokenserver → ui
// incremental assistant text — append in order
{ "type": "token", "content": "Pulling Q2 numbers" }
tool_stepserver → ui
// start: open the step card. end: fill output + status
{ "type": "tool_step",
  "id": "step_a1",
  "name": "query_warehouse",
  "phase": "start",            // "start" | "end"
  "input": { "sql": "select ..." },
  "output": null,                // set on phase:"end"
  "status": "running" }          // running | ok | error
approval_requestserver → ui
// graph paused at interrupt_before — render an approval card
{ "type": "approval_request",
  "id": "appr_7c",
  "tool": "run_python",
  "input": { "code": "import pandas ..." },
  "reason": "Executes code in the sandbox",
  "editable": true }            // can the user tweak input before approving
POST /approvals/{id}ui → server
// resolves the pause; graph resumes from checkpoint
{ "decision": "approve",        // "approve" | "reject"
  "edited_input": { "code": "..." }, // optional, when editable
  "note": "trimmed the date range" }  // optional, audit trail
doneserver → ui
{ "type": "done",
  "message_id": "msg_42",
  "finish_reason": "stop" }       // stop | rejected | length
errorserver → ui
{ "type": "error",
  "message": "warehouse timeout",
  "recoverable": true }          // ui offers retry when true

06 Per-skill approval gates
