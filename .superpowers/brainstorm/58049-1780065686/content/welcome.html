Designer Canvas — what I found in the code

Grounding the PRD against the actual repo before we refine it

Dynadoc (feat/dynadoc branch)

render_tree(node, load_json, load_template, strict) already renders any Node subtree — the engine is recursive over a manifest tree. The public render_document(name) only exposes whole-document rendering, but single-section render is a thin new entry point, not a rewrite. (Answers Open Q1 optimistically.)

Identity gap

Manifest Node.path is a structural dotted path (lld.data.models). The PRD's identity (D6) is the natural key inside the JSON (OverdraftCheck). Dynadoc emits pure Markdown today — no node-key marker. So the marker (Open Q5) is genuinely new work in the template/wrapper layer.

Event stream

Today the frontend = Chainlit consuming raw LangGraph astream_events: on_chat_model_stream, on_tool_start/end, on_chain_end (carries structured output). The PRD's semantic events (section_updated, gate_changed, stage_changed) don't exist yet — they're a new layer derived from the raw stream.

Agent topology ↔ sections

Designer's agents.yaml confirms the mapping: coordinator, background_and_scope, data_models, services… each owns a schema + section. The PRD's "outline = pipeline" claim holds.
