Option 1 · Buttons emit structured events directly
User clicks Approve on OverdraftCheck
│ structured control event
Backend control endpoint
{type: gate_approve, node: OverdraftCheck}
│ flips gate, advances coordinator
State updated → gate_changed pushed → rail/section re-render
⋮ (chat untouched)
Chat stays clean — only NL turns appear here

Pros

  • Clean separation: control vs conversation
  • No NL parsing of approvals — deterministic
  • Chat transcript isn't polluted with "approve" noise

Cons

  • Two input channels to the backend (chat + control)
  • D2 reworded: "no freeform content editing" (buttons OK)
Option 3 · Buttons drop a message into chat
User clicks Approve on OverdraftCheck
│ injects templated turn
Chat receives a structured message
"✓ Approve OverdraftCheck" (machine-tagged)
│ single input channel
Coordinator consumes the turn → flips gate
│
State updated → gate_changed pushed → re-render

Pros

  • One conceptual input channel (everything is a turn)
  • Full audit trail — every decision is in the transcript
  • Click ergonomics, chat semantics

Cons

  • Transcript fills with approve/reject turns
  • Coordinator must reliably parse tagged turns → still deterministic only if tagged, not free NL
  • Round-trip through the LLM for what could be a pure state flip

My lean: 1. A gate flip is a deterministic state change; sending it through the LLM (option 3) re-introduces exactly the "every correction routes back through the prompt box" friction the PRD calls out as a problem. Keep chat for intent, controls for decisions — and log control actions to the same history so you still get the audit trail.
