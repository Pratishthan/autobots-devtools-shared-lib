Frontend stack: server-driven vs React SPA

The canvas is a pure function render(state)→view. State is server-authoritative, frontend never mutates. Click an option when you're ready.

A

Server-driven (HTMX / Datastar + WebSocket)

Backend owns rendering. On each section_updated event, the server re-renders that section's Dynadoc Markdown→HTML fragment and pushes it over WS/SSE. The browser swaps the fragment in by data-node-id. Thin JS only for click-to-pin and chat input.

B

React SPA + FastAPI WebSocket

FastAPI pushes structured JSON events. A React store reduces over them; components re-render. Dynadoc Markdown is rendered client-side (or server-pre-rendered HTML injected). Rich client state, virtual DOM diffing.

A · Server-driven

Pros

  • One language. Stays in Python/Jinja — same world as Dynadoc templates & today's Chainlit
  • No SPA build/deploy pipeline, no node_modules, no second CI lane
  • render(state)→view is literal — server is the single renderer; no client/server state to keep in sync
  • Reuses Dynadoc HTML output directly — fragment per node maps 1:1 to the swap unit
  • Reconnection = re-request current state & re-render; trivially correct

Cons

  • Rich client interactions (drag, smooth diff animations, complex impact-panel graph) are harder
  • Smaller talent/ecosystem vs React; fewer off-the-shelf components
  • Every interaction is a round-trip (fine for single-user, server-authoritative)
B · React SPA

Pros

  • Richest interaction ceiling — graph views, animations, optimistic UI
  • Huge ecosystem (component libs, diff viewers, graph viz)
  • Clear reducer model matches "frontend = reducer over event stream" framing
  • Natural home if Nurture/KBE later want heavier client UIs

Cons

  • Second language + toolchain the team doesn't run today (build, bundle, deploy, version)
  • Two copies of state (server truth + client store) — the sync bugs D2 was designed to avoid
  • Markdown→identity markers must survive into client render; more plumbing
  • Higher baseline effort before first pixel; overkill for read-only + single user

My lean: A (server-driven) — it's the smallest stack that fully honors D2/D3 and keeps the team in one language. The only panel where React clearly wins is the impact-panel graph, and that can be an embedded JS island (e.g. a small graph lib) inside a server-driven shell. Which way do you want to go?
